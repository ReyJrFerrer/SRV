# Notification System

Multi-channel notification system using Firestore as the notification store, OneSignal for push delivery, and Firebase Cloud Functions for transactional email. Covers booking lifecycle, chat messages, reviews, payments, and system announcements.

## Backend: `functions/src/notification.js`

### Entrypoint

```
notificationAction ({ action, data }) → dispatches to internal handler
```

#### Notification Statuses

```
NOTIFICATION_STATUS =
  unread          — created but not yet pushed or read
  read            — marked as read (from unread)
  push_sent       — push notification delivered (unread)
  push_sent_and_read — push delivered and read
```

### Firestore Schema

```
notifications/{notificationId}
  id: string                   — auto-generated by Firestore
  userId: string
  userType: "client" | "provider"
  notificationType: string     — one of NOTIFICATION_TYPES values
  title: string
  message: string
  relatedEntityId: string|null  — booking ID, conversation ID, etc.
  metadata: object|null         — arbitrary data (e.g. senderId, senderName, customHref)
  href: string                 — generated by generateNotificationHref()
  status: string               — one of NOTIFICATION_STATUS values
  createdAt: Timestamp         — server timestamp
  readAt: Timestamp|null
  pushSentAt: Timestamp|null
  expiresAt: Timestamp         — createdAt + 30 days

notificationFrequency/{userId}_{notificationType}
  userId: string
  notificationType: string
  timestamps: number[]         — millisecond timestamps within SPAM_PREVENTION_WINDOW
  lastUpdated: Timestamp

chatEmailCooldowns/{receiverId}_{conversationId}
  receiverId: string
  conversationId: string
  lastEmailSentAt: Timestamp
```

### Actions

| Action | Auth | Description |
|---|---|---|
| `createNotification` | Authenticated | Creates a notification doc after spam check (`isSpamming`). Generates href via `generateNotificationHref()`. Sends OneSignal push (non-blocking, fire-and-forget). If type is in `BOOKING_EMAIL_TYPES`, sends email (non-blocking). Updates `notificationFrequency` for spam tracking. |
| `getUserNotifications` | Owner/Admin | Queries `notifications` collection with optional filters: `userType`, `notificationType`, `status`, `limit`. Converts Firestore Timestamps to ISO strings. |
| `markNotificationAsRead` | Owner/Admin | Transactional update: sets status to "read" or "push_sent_and_read" (preserving push_sent status), sets `readAt`. |
| `markNotificationAsPushSent` | Owner/Admin | Transactional update: sets status to "push_sent" or "push_sent_and_read" (preserving read status), sets `pushSentAt`. |
| `getNotificationsForPush` | Owner/Admin | Returns up to 50 unread notifications for push delivery. |
| `storeOneSignalPlayerId` | Authenticated | Validates UUID format (rejects `local-` prefixed dev IDs). Adds to `users/{uid}.oneSignalPlayerIds[]` via `arrayUnion`. Creates user doc if not exists. |
| `removeOneSignalPlayerId` | Authenticated | Removes specific player ID via `arrayRemove`, or clears all if no ID provided. |
| `getNotificationStats` | Owner/Admin | Counts by status: total, unread (±push_sent), pushSent, read. Returns `{ total, unread, pushSent, read }`. |
| `markAllNotificationsAsRead` | Authenticated | Batch marks all `unread` and `push_sent` notifications as read/`push_sent_and_read`. Returns count. |
| `canReceiveNotification` | Authenticated | Returns boolean by checking `isSpamming()` for the given userId + notificationType. |
| `deleteNotification` | Owner/Admin | Hard-deletes the notification document from Firestore. |

### Notification Href Generation

`generateNotificationHref(notificationType, userType, entityId)` maps types to frontend routes:

| Type | Client Route | Provider Route |
|---|---|---|
| `chat_message` | `/client/chat/{entityId}` | `/provider/chat/{entityId}` |
| `review_reminder` / `review_request` | `/client/review/{entityId}` | `/provider/rate-client/{entityId}` |
| `payment_completed` | — | `/provider/receipt/{entityId}` |
| `start_service` / `service_completion_reminder` | `/client/booking/{entityId}` | `/provider/active-service/{entityId}` |
| `start_navigation` | `/client/booking/{entityId}` | `/provider/directions/{entityId}` |
| `booking_auto_cancelled_*` | `/client/home` | `/provider/bookings` |
| All others | `/client/booking/{entityId}` | `/provider/booking/{entityId}` |

A `metadata.customHref` overrides the generated href.

### OneSignal Push Delivery

`sendOneSignalNotification(userId, notification)`:

1. Fetches `users/{userId}` doc for `oneSignalPlayerIds` array
2. Filters to valid UUIDs (rejects non-UUID, `local-` prefixes)
3. If no valid player IDs, returns early
4. POSTs to `https://onesignal.com/api/v1/notifications` with:
   - `app_id` (from `ONESIGNAL_APP_ID` env)
   - `include_player_ids` (all valid player IDs for multi-device support)
   - `headings` / `contents` from notification title/message
   - `data` payload with `notificationId`, `type`, `relatedEntityId`, `metadata`, `href`
   - `url` for click-through: `{APP_BASE_URL}/#{hrefPath}`
5. On success, updates notification status to `push_sent`

Push is fire-and-forget (`.catch()`). Failures are logged but not retried.

### Email Notification Delivery

`sendEmailForNotification(userId, notification)`:

1. Fetches user's email from `users/{userId}.email`
2. For booking-related notifications, calls `buildBookingDetailsForEmail()` to enrich with:
   - Service name, package names
   - Scheduled date/time (with Asia/Manila timezone)
   - Location with Google Maps link
   - Price in PHP (₱) format
   - Other party's contact info (name, phone, email)
3. For chat messages, fetches sender info (name, phone, role)
4. Builds HTML+text email via `buildEmailTemplate()` utility
5. Sends via `sendEmail()` utility

Email is fire-and-forget with `.catch()`. Chat emails are additionally rate-limited to 1/hour per conversation pair via `chatEmailCooldowns` collection.

### Spam Prevention

`isSpamming(userId, notificationType)` checks `notificationFrequency/{userId}_{notificationType}`:
- If ≥10 timestamps within the last 5-minute window, returns true (reject)
- `updateNotificationFrequency()` prunes stale timestamps and appends current timestamp
- `createNotification` throws `resource-exhausted` if spamming

### Other Sources of Notifications

Notifications are created not only via `notificationAction("createNotification")` but also from:

| Source | File | How |
|---|---|---|
| Chat messages | `chat.js:126-153` | `onMessageCreated` trigger creates notification + OneSignal push + email with 1h cooldown |
| Booking state changes | `booking.js` | Various booking status transitions call notification creation directly |
| Review submissions | `review.js` | Via `processReviewForReputationInternal` -> `createNotification` |
| Cancellation reports | `booking.js` | Creates report with cancellation notification |

### Scheduled Functions

```
cleanupExpiredNotifications
  Schedule: daily at midnight (0 0 * * *)
  Deletes notification docs where expiresAt <= now

cleanupNotificationFrequency
  Schedule: every 6 hours (0 */6 * * *)
  Prunes stale timestamps from notificationFrequency collection,
  deletes docs with no recent timestamps
```

## Frontend Hooks

### `useNotificationsWithPush` (`hooks/useNotificationsWithPush.ts`)

Client notification hook (554 lines):
- Subscribes to Firestore notifications via `notificationCanisterService.subscribeToUserNotifications()`
- Integrates with OneSignal via `notificationIntegrationService.initialize()`
- Maintains in-memory `notificationStore` for cross-instance unread count
- Global event system (`MARK_AS_READ_EVENT`, `MARK_ALL_AS_READ_EVENT`) for coordinating between hook instances
- Dual storage: reads/writes to Firestore via canister, falls back to localStorage
- Tracks previous notification IDs to detect new notifications for sound/re-badge

### `useProviderNotificationsWithPush` (`hooks/useProviderNotificationsWithPush.ts`)

Provider-specific variant (549 lines):
- Same architecture as client hook but with provider-specific notification types
- Filters to `userType: "provider"`
- Provider unread count in separate `providerNotificationStore`
- Integrates with `useProviderBookingManagement` for booking action dispatch from notifications

### `useChatNotifications` (`hooks/useChatNotifications.tsx`)

Chat-specific notification state (226 lines):
- Tracks unread conversations count
- Plays notification sound on new messages
- Uses custom DOM events (`chats-read`, `conversations-updated`, `messages-updated`)

### `usePWA` (`hooks/usePWA.ts`)

PWA state hook (437 lines):
- Push notification support detection
- Subscription state management (subscribed/unsubscribed)
- Permission handling (granted/denied/default)
- Install prompt event handling

## Phase 1 — Online Services Notification Types (8 new)

Phase 1 of the Online Services rollout adds 8 new notification types for the `OnlineProject` lifecycle. Each follows the same dispatch pattern as booking notifications: a local `createNotification()` call inside `onlineProject.js` after the state transition is committed.

| # | Type Constant | Trigger Action | Recipient | Href (client) | Href (provider) |
|---|---------------|----------------|-----------|---------------|-----------------|
| 1 | `project_created` | `createOnlineProject` | Provider | — | `/provider/project/{id}` |
| 2 | `project_accepted` | `acceptProject` | Client | `/client/project/{id}` | — |
| 3 | `project_declined` | `declineProject` | Client | `/client/project/{id}` | — |
| 4 | `project_negotiation_received` | `negotiateProject` (either side) | Other party | `/client/project/{id}?tab=negotiations` | `/provider/project/{id}?tab=negotiations` |
| 5 | `project_negotiation_accepted` | `acceptCounterOffer` | Other party | `/client/project/{id}` | `/provider/project/{id}` |

| 7 | `deliverable_approved` | `approveDeliverable` (full or partial) | Provider | — | `/provider/project/{id}?tab=deliverables` |
| 8 | `revision_requested` | `requestRevision` | Provider | — | `/provider/project/{id}?tab=deliverables` |

### Dispatch Pattern

Each action in `onlineProject.js` calls the local `createNotification()` after the Firestore transaction commits. The pattern:

```javascript
// Inside onlineProjectAction switch case
await createNotification({
  targetUserId: otherPartyId,
  userType: otherPartyRole,        // 'client' | 'provider'
  type: 'project_created',
  title: 'New project request',
  message: `${clientName} requested "${project.title}"`,
  relatedEntityType: 'online_project',
  relatedEntityId: project.id,
  metadata: { projectId: project.id, clientId, providerId },
});
```

`createNotification()` (in `notification.js`) handles the spam-prevention check (`canReceiveNotification`), OneSignal push dispatch, and email dispatch.

### Multi-Session Booking Notifications (Phase 2)

Reuse existing booking notification types (`BOOKING_*`). No new types needed for Phase 2 multi-session booking. A future enhancement may add `SESSION_REMINDER` (24h before each session) but is deferred.
